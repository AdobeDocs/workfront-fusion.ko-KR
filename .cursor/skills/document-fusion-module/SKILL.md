# Fusion 커넥터 모듈 문서화

이 스킬을 사용하여 Workfront Fusion 커넥터 문서에 하나 이상의 새 모듈을 추가합니다(예: `adobe-firefly-modules.md`, `adobe-photoshop-modules.md`, `azure-dev-ops.md`).

## 필수 입력(모듈당)

모듈의 필드를 채우려면 먼저 **must** 사용자가 다음을 모두 제공해야 합니다. 누락된 항목이 있으면 중지했다가 계속 요청하십시오.

1. **모듈 이름** — Fusion UI에 표시되는 것과 정확히 동일합니다(예: `Generate adaptive composite`).
2. **커넥터 문서 경로** — 에이전트가 이 경로를 찾아 사용자에게 확인합니다(1단계 참조).
3. 기본 API 작업에 대한 **API URL**. API 사양에 대해 각 필드의 목적을 상호 참조할 수 있도록 이 작업이 필요합니다.
4. **표준 보기에서 모듈 구성 패널의 스크린샷**(`Show advanced settings` **선택되지 않음**).
5. **고급 보기에서 모듈 구성 패널의 스크린샷**(`Show advanced settings` **확인**) 또는 모듈에 고급 필드가 없다는 명시적 문입니다.

## 워크플로

이는 다단계 프로세스입니다. 간격 지정이 중요합니다. *사용자를 사용하여* 작업하십시오. 앞서 작업하지는 마십시오. 각 단계 경계에서 진행 상황을 확인하고 언제든지 수정할 수 있습니다.

### 1단계 — 작업 범위 지정

1. **모듈이 한 개인지 아니면 여러 개인지 묻기**: *&quot;커넥터 문서에 단일 모듈 또는 여러 모듈을 추가하시겠습니까?&quot;*

2. **답변을 토대로 모든 모듈 이름 수집**:
   - **단일 모듈**: Fusion UI에 표시되는 정확한 이름을 지정하십시오.
   - **여러 모듈**: 모든 이름을 한 번에 요청하거나 이러한 이름을 나열하는 커넥터의 스크린샷을 요청합니다(예: 각 모듈에 대한 타일 레이블을 표시하는 Fusion의 커넥터 개요 화면). 어느 경우든 추가할 모든 새 모듈의 확정된 목록을 사용하여 이 단계를 종료합니다.

3. **모듈에서 암시하는 커넥터 이름을 기반으로 `help/workfront-fusion/references/apps-and-modules/`을(를) 검색하여 직접 커넥터 문서를 찾습니다**. `Glob` 또는 `Grep` 도구를 사용합니다.

4. **사용자와 문서 경로를 확인**: *&quot;모듈 이름을 기반으로 `help/workfront-fusion/references/apps-and-modules/adobe-connectors/adobe-firefly-modules.md`에 입력해야 합니다. 맞습니까?&quot;*

5. **커넥터 문서를 읽고**&#x200B;기존 구조 및 규칙을 알아봅니다. 다음 사항에 주의하십시오.
   - 제목 스타일(일반적으로 `### Module name`(문장의 경우)
   - 기존 모듈 순서 지정(일반적으로 알파벳순)
   - `Connection`개 행의 문구 표시 방법
   - 중첩된 필드 작성 방법(예: `Image > Source`)
   - 기존의 각주 또는 주석 규칙

### 2단계 - 모든 모듈의 자리 표시자를 맨 앞에 배치

새 모듈이 하나뿐이라도 사용자에게 명확한 시각적 로드맵을 제공합니다. 새 모듈이 여러 개 있는 경우 모든 모듈을 지금 배치하십시오.

각 새 모듈에 대해 자리 표시자 섹션을 알파벳순으로 삽입합니다.

```markdown
### Module name

<!-- PLACEHOLDER: Add description of the Module name module here. -->

<table style="table-layout:auto"> 
 <col> 
 <col> 
 <tbody> 
  <tr> 
   <td role="rowheader">[!UICONTROL Connection]</td> 
   <td>For instructions on creating a connection to [!DNL Connector Name], see <a href="#create-a-connection-to-connector-name" class="MCXref xref" >Create a connection to [!DNL Connector Name]</a> in this article.</td> 
  </tr> 
  <!-- PLACEHOLDER: Add field rows for the Module name module here. -->
 </tbody> 
</table>
```

추가된 자리 표시자 목록을 사용자에게 표시합니다(예: &quot;에 대해 추가된 자리 표시자: 적응형 합성 생성, 이미지5를 사용하여 이미지 생성, 정확한 합성 생성, 비디오 생성&quot;). 다음으로 시작할 항목을 묻습니다. *&quot;어떤 항목으로 시작하시겠습니까?&quot;*

### 3단계 — 모듈별 설명서 루프

각 모듈에 대해 다음 모듈로 이동하기 전에 이 루프를 완전히 완료하십시오.

#### 3a. API URL 가져오기

사용자에게 API URL을 요청하십시오. *&quot;`<module name>`의 기본 API 작업에 대한 API URL을 공유하십시오.&quot;*

보유하고 있는 경우:
- URL에서 `WebFetch`을(를) 시도하십시오.
- 페이지가 비어 있거나 JS가 렌더링된 경우(Adobe Developer 문서에서 공통됨) 관련 기능 안내서 페이지(일반적으로 `.../guides/how-tos/...` URL에서)를 찾아 대신 가져옵니다.
- 그래도 문제가 해결되지 않으면 작업 이름의 `WebSearch`(으)로 돌아가십시오.

나중에 필드 설명을 위한 배경 컨텍스트로 학습한 내용을 사용합니다. 사용자에게 덤프하지 말고 간단히 인식하십시오. *&quot;API 컨텍스트를 가져왔습니다. 표준 보기 스크린샷에 사용할 준비가 되었습니다.&quot;*

#### 3b. 표준 뷰 스크린샷 가져오기

질문: *&quot;모듈 구성 패널의 스크린샷을 `Show advanced settings`**선택 취소됨**과(와) 공유하십시오. 표준 필드를 연습한 후 고급 보기를 요청하지만 아직 보내지 않습니다.&quot;*

#### 3c. 표준 필드 문서화

표준 뷰 스크린샷에서 작업하여 보이는 모든 필드를 위에서 아래로 캡처합니다. 각 필드에 대해:

1. 필드의 정확한 UI 레이블을 행 머리글로 사용합니다. 그룹화된 필드의 경우 ` > `(으)로 조인:

   ```
   [!UICONTROL Background > Image > Source]
   ```
2. 스크린샷(회색/노란색 캡션)의 필드 아래에 있는 도우미 텍스트를 읽습니다. 설명을 위한 기반으로 사용하십시오.
3. 3a단계의 API 컨텍스트를 상호 참조하여 필드가 나타내는 영역을 식별합니다(예: `background.fillAreaMask`은(는) &quot;개체가 배치될 배경의 영역&quot;).
4. **필드 내용**(API에서) 및 **제공 방법**(UI 도우미 텍스트 및 필드 유형에서)을 결합하여 설명을 작성하십시오.

API가 지원하는 경우에도 스크린샷에 표시되지 않는 필드를 건너뜁니다. 밭을 만들지 마라.

표준 필드가 갖추어지면 사용자에게 이를 간단히 요약한 후(예: &quot;추가된 표준 필드: 연결, 프롬프트, 종횡비, ...&quot;) 3d단계로 이동합니다.

#### 3d. 고급 뷰 스크린샷 가져오기

질문: *&quot;이제 모듈 구성 패널의 스크린샷을 `Show advanced settings`**확인**과(와) 공유하십시오. 모듈에 고급 필드가 없으면 그렇게 하면 건너뜁니다.&quot;*

#### 3e. 고급 필드 문서화

고급 뷰 스크린샷에서 작업:

1. *이(가) 이미 표준 보기에 없는* 필드를 식별합니다. 이 필드는 고급 필드입니다.
2. 각 고급 필드에 대해 3c단계의 동일한 설명 프로세스를 따릅니다.
3. `<td role="rowheader">` 내의 필드 이름에 `*` 추가:

   ```html
   <td role="rowheader">[!UICONTROL Seeds]*</td>
   ```
4. `</table>`을(를) 닫은 후 자신의 줄에 이 각주를 추가하십시오.

   ```markdown
   * These fields are advanced fields, and do not display unless you select **[!UICONTROL Show advanced settings]**.
   ```

모듈에 고급 필드가 없다고 표시되는 경우 3d 및 3e 단계를 완전히 건너뜁니다.

#### 3f. 모듈 설명 초안 작성

이제(이전은 아님) 섹션 맨 위에 있는 설명 자리 표시자를 API 컨텍스트와 필드에 포함된 내용을 기반으로 한 한 문장 초안으로 바꿉니다. 항상 사용자의 검토를 위한 초안으로 제공합니다.

> *&quot;이 설명을 작성했습니다. &#39;..&#39; 사용하거나 편집하거나 바꾸시겠습니까?&quot;*

#### 3g. 모듈별 요약

사용자에게 모듈의 문서화된 필드(표준 + 고급)에 대한 최종 요약을 제공하고 열려 있는 질문을 표시하십시오.

- 스크린샷에서 잘린 필드가 있었나요?
- 모듈 설명이 허용됩니까?
- 더 이상 사용되지 않거나 메모할 가치가 있는 것으로 플래그가 지정되어야 합니까?

모듈이 완료되었다고 말하기 전에 확인을 기다립니다. 다음 질문하기: *&quot;다음 모듈로 이동할 준비가 되셨습니까?&quot;* (또는 &quot;이 커넥터 문서 작업이 모두 완료되었습니다.&quot;)

### 4단계 — 최종 확인

모든 모듈이 문서화되면 각 모듈에 대해 이 체크리스트를 실행합니다.

- [ ] 모듈 머리글은 `### `(H3)이며 대소문자를 사용합니다.
- [ ] 모듈이 형제 모듈에 대한 알파벳 순서로 나타납니다.
- [ ] 첫 행은 `Connection`이며 표준 연결 링크 단어를 사용합니다.
- [ ] 모든 문서화된 필드가 사용자의 스크린샷에 표시됩니다.
- [ ] 고급 필드가 `*`(으)로 표시되어 있고 표 아래에 하나의 각주가 있습니다.
- [ ] API에서만 만들어진 필드가 없습니다.
- [ ] Source 드롭다운 설명에서 정확한 UI 옵션 레이블을 사용합니다(아래 &quot;Source 필드 규칙&quot; 참조).
- [ ] 모듈 설명은 모듈의 작업을 설명하는 한 문장 요약입니다.

## Source 전당 대회

Fusion 커넥터의 이미지 소스 필드는 일반적으로 두 가지 옵션이 있는 드롭다운을 표시합니다. **사용자의 스크린샷에 표시된 정확한 레이블을 사용합니다** — 세대의 모듈에 따라 다른 단어가 사용됩니다.

| 드롭다운에 다음이 표시되는 경우... | 이 형식 사용 |
|---|---|
| **이미지 업로드** 및 **이미지 URL**(최신 모듈) | `<ul><li><b>Upload Image</b><p>Upload the image, or map the image file from a previous module.</p></li><li><b>Image URL</b><p>Enter or map the URL of the image.</p></li></ul>` |
| **파일** 및 **사전 서명된 URL**(이전 모듈) | `<ul><li><b>File</b><p>Select a source file from a previous module, or map the source file's reference image file name and reference image file.</p></li><li><b>Presigned URL</b><p>Enter or map the URL of the source image.</p></li></ul>` |

스크린샷에 드롭다운 옵션이 열리지 않으면 사용자에게 공유하도록 요청하십시오.

## 필드 설명 스타일

- 동일한 문서에서 기존 모듈 항목의 음성 및 구조를 일치시키십시오.
- 설명을 짧게 유지하십시오. 일반적으로 필드당 한두 문장이면 충분합니다.
- 다른 필드에서 참조한 UI 필드 이름에 `[!UICONTROL ...]` 태그를 사용합니다.
- 리터럴 값(`<code>en-US</code>`, `<code>0.0</code>`)에 `<code>...</code>`을(를) 사용합니다.
- 숫자 범위 필드의 경우 허용 범위를 명시적으로 지정하십시오. *&quot;0과 1 사이의 숫자를 입력하십시오...&quot;*.
- 개별 옵션이 있는 드롭다운의 경우 `<b>Option</b>` 다음에 `<p>` 설명이 있는 `<ul>`(으)로 열거하십시오.

## 공통 필드 및 권장 설명

모듈 간 일관성을 유지하기 위해 재사용:

**시드**(항목 추가를 통해 필드에서 여러 시드 값을 허용하는 경우):
> **항목 추가**&#x200B;를 클릭하여 시드 값을 추가한 다음 정수를 입력하거나 매핑합니다. 변형당 하나의 시드 사용. 시드 값 수는 둘 다 제공된 경우 **변형 수** 값과 일치해야 합니다.

**변형 수**(스크린샷의 실제 범위로 범위 바꾸기):
> [min]에서 [max] 사이의 숫자를 입력하세요. 모듈이 이 개수의 [출력 형식] 변형을 생성합니다.

**제한**(표준 Fusion 실행 주기 제한, 스크린샷에 표시되는 경우에만 포함):
> 한 실행 주기 동안 모듈에서 작업할 최대 결과 수를 입력하거나 매핑합니다.

**로케일**:
> 언어 및 지역 코드를 입력하거나 매핑하여 생성된 콘텐츠를 특정 국가 및 언어로 사용자 지정합니다. 로케일은 ISO 639-1 언어 코드 및 ISO 3166-1 지역에서 제공해야 합니다. 예: `en-US`

## 언제든지 수정할 수 있음

초기 작업 미드플로우(시드 문구, 드롭다운 레이블, 필드 위치 지정, 고급/표준 분할 등)를 수정할 수 있습니다. 모든 수정 사항을 신뢰할 수 있는 것으로 취급하고 푸시백 없이 업데이트합니다. 수정 후 변경된 사항을 간단히 정리합니다.

사용자가 새 규칙 미드플로우를 도입하는 경우(예: &quot;고급 필드에 `*`(으)로 표시하고 각주를 추가하겠습니다&quot;) 나머지 세션에 이 규칙을 채택하고 이미 문서화된 모듈에 소급하여 적용합니다.

## 정보가 충돌하는 경우 수행할 작업

API 사양 및 UI가 동의하지 않는 경우(종종 발생):

1. **UI를 신뢰**. 사용자가 실제로 Fusion에서 볼 수 있는 내용이기 때문입니다.
2. **추가 조사 여부를 결정할 수 있도록 사용자에게 보낸 회신에 불일치점을 적어 두십시오**.
3. **문서 본문에 설명을 만들지 마십시오** — UI에 있는 내용을 사실대로 유지하십시오.
